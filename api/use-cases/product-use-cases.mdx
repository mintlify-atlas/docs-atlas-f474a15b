## Overview

Product use cases define the business operations available for managing products in the inventory system. Following hexagonal architecture principles, these use cases represent the application's core business logic, independent of frameworks and infrastructure.

<Note>
  Use cases are defined as interfaces in the core layer and implemented by service classes. This approach supports dependency inversion and testability.
</Note>

## Architecture Context

Use cases sit between the REST adapters (input ports) and repositories (output ports):

```
REST Controller → Use Case Interface → Service Implementation → Repository
```

## RegisterProductUseCase

Handles the registration of new products in the system.

### Interface

```java
// Source: core/src/main/java/com/fbaron/ims/product/usecase/RegisterProductUseCase.java

package com.fbaron.ims.product.usecase;

import com.fbaron.ims.product.model.Product;

public interface RegisterProductUseCase {

    Product register(Product product);

}
```

### Method: register()

<ParamField path="register" type="method">
  Registers a new product in the inventory system.
  
  **Signature:** `Product register(Product product)`
</ParamField>

#### Parameters

<ParamField path="product" type="Product" required>
  The product domain model to register. Should contain:
  - `name` - Product name (required)
  - `price` - Product price (required, > 0.01)
  - `description` - Product description (optional)
  - `category` - Product category (required)
  
  The `id` field should be null as it will be auto-generated.
</ParamField>

#### Returns

<ResponseField name="Product" type="Product">
  The registered product with the generated `id` field populated.
</ResponseField>

#### Business Rules

<CardGroup cols={2}>
  <Card title="ID Generation" icon="key">
    The product ID (UUID) is automatically generated by the persistence layer. Input products should not have an ID set.
  </Card>
  
  <Card title="Validation" icon="shield-check">
    Input validation occurs at the DTO layer before reaching this use case. The use case assumes valid input.
  </Card>
  
  <Card title="Uniqueness" icon="fingerprint">
    Currently, no uniqueness constraints are enforced on product names. Multiple products can have the same name.
  </Card>
  
  <Card title="Persistence" icon="database">
    The product is immediately persisted to the database. This operation is not transactional across multiple products.
  </Card>
</CardGroup>

### Implementation

```java
// Source: core/src/main/java/com/fbaron/ims/product/service/ProductService.java

@RequiredArgsConstructor
public class ProductService implements RegisterProductUseCase, GetProductUseCase {

    private final ProductQueryRepository productQueryRepository;
    private final ProductCommandRepository productCommandRepository;

    @Override
    public Product register(Product product) {
        return productCommandRepository.save(product);
    }
    
    // ... other methods
}
```

#### Implementation Notes

<Expandable title="Why So Simple?">
  The implementation is straightforward because:
  
  1. **Validation** happens at the DTO/REST layer using Bean Validation
  2. **ID Generation** is handled by the persistence layer
  3. **Business Rules** for products are currently minimal
  
  As business requirements grow (e.g., duplicate checking, SKU generation, supplier integration), this implementation would expand while the interface remains stable.
</Expandable>

### Usage Example

```java
// In ProductRestAdapter
@PostMapping
public ResponseEntity<ProductDto> registerProduct(
    @Valid @RequestBody RegisterProductDto dto) {
    
    // Map DTO to domain model
    var product = productDtoMapper.toModel(dto);
    
    // Execute use case
    var registeredProduct = registerProductUseCase.register(product);
    
    // Map domain model back to DTO and return
    return ResponseEntity.status(HttpStatus.CREATED)
            .body(productDtoMapper.toDto(registeredProduct));
}
```

### Exception Handling

Potential exceptions (though not explicitly thrown in current implementation):

- **ConstraintViolationException** - Database constraint violation
- **DataAccessException** - Database connectivity or persistence issues

## GetProductUseCase

Handles retrieval operations for products.

### Interface

```java
// Source: core/src/main/java/com/fbaron/ims/product/usecase/GetProductUseCase.java

package com.fbaron.ims.product.usecase;

import com.fbaron.ims.product.model.Product;
import java.util.List;
import java.util.UUID;

public interface GetProductUseCase {

    Product getById(UUID id);

    List<Product> getAll();

}
```

### Method: getById()

<ParamField path="getById" type="method">
  Retrieves a single product by its unique identifier.
  
  **Signature:** `Product getById(UUID id)`
</ParamField>

#### Parameters

<ParamField path="id" type="UUID" required>
  The unique identifier of the product to retrieve.
</ParamField>

#### Returns

<ResponseField name="Product" type="Product">
  The product with the specified ID.
</ResponseField>

#### Throws

<ResponseField name="ProductNotFoundException" type="RuntimeException">
  Thrown when no product exists with the specified ID.
  
  **Message:** `"Product not found with ID: {id}"`
  
  **HTTP Status:** 404 Not Found
</ResponseField>

### Method: getAll()

<ParamField path="getAll" type="method">
  Retrieves all products in the system.
  
  **Signature:** `List<Product> getAll()`
</ParamField>

#### Returns

<ResponseField name="List<Product>" type="List<Product>">
  A list of all registered products. Returns an empty list if no products exist.
</ResponseField>

<Warning>
  This method loads all products without pagination. For systems with large numbers of products, consider implementing pagination to avoid performance issues.
</Warning>

### Implementation

```java
// Source: core/src/main/java/com/fbaron/ims/product/service/ProductService.java

@RequiredArgsConstructor
public class ProductService implements RegisterProductUseCase, GetProductUseCase {

    private final ProductQueryRepository productQueryRepository;
    private final ProductCommandRepository productCommandRepository;

    @Override
    public List<Product> getAll() {
        return productQueryRepository.findAll();
    }

    @Override
    public Product getById(UUID id) {
        return productQueryRepository.findById(id)
                .orElseThrow(() -> new ProductNotFoundException(id));
    }
}
```

#### Implementation Notes

<Expandable title="Error Handling Strategy">
  The `getById()` method uses Java's `Optional.orElseThrow()` pattern:
  
  1. Repository returns `Optional<Product>`
  2. If present, product is returned
  3. If empty, `ProductNotFoundException` is thrown
  4. Exception is caught by Spring's exception handler
  5. Converted to HTTP 404 response
  
  This approach avoids null checks and provides clear error messages.
</Expandable>

  <Tab title="Get by ID">

## Exception Details

### ProductNotFoundException

```java
// Source: core/src/main/java/com/fbaron/ims/product/exception/ProductNotFoundException.java

package com.fbaron.ims.product.exception;

import java.util.UUID;

public class ProductNotFoundException extends RuntimeException {

    public ProductNotFoundException(UUID productId) {
        super("Product not found with ID: " + productId);
    }
}
```

<ParamField path="ProductNotFoundException" type="exception">
  Domain exception thrown when a product cannot be found.
  
  **Extends:** `RuntimeException` (unchecked exception)
  
  **When Thrown:**
  - `getById()` called with non-existent product ID
  - Inventory operations reference non-existent product
  
  **HTTP Mapping:** 404 Not Found
</ParamField>

## Repository Interfaces

Use cases depend on repository interfaces (output ports):

### ProductQueryRepository

```java
public interface ProductQueryRepository {
    Optional<Product> findById(UUID id);
    List<Product> findAll();
}
```

<Note>
  Query repository handles all read operations following CQRS principles.
</Note>

### ProductCommandRepository

```java
public interface ProductCommandRepository {
    Product save(Product product);
}
```

<Note>
  Command repository handles all write operations following CQRS principles.
</Note>

## Testing Use Cases

Use cases can be tested independently of infrastructure:

```java
class ProductServiceTest {

    @Mock
    private ProductQueryRepository queryRepository;
    
    @Mock
    private ProductCommandRepository commandRepository;
    
    @InjectMocks
    private ProductService productService;

    @Test
    void shouldRegisterProduct() {
        // Given
        Product product = Product.builder()
            .name("Test Product")
            .price(new BigDecimal("99.99"))
            .category("Test")
            .build();
            
        Product savedProduct = Product.builder()
            .id(UUID.randomUUID())
            .name("Test Product")
            .price(new BigDecimal("99.99"))
            .category("Test")
            .build();
            
        when(commandRepository.save(product)).thenReturn(savedProduct);
        
        // When
        Product result = productService.register(product);
        
        // Then
        assertNotNull(result.getId());
        assertEquals("Test Product", result.getName());
        verify(commandRepository).save(product);
    }
    
    @Test
    void shouldThrowExceptionWhenProductNotFound() {
        // Given
        UUID nonExistentId = UUID.randomUUID();
        when(queryRepository.findById(nonExistentId))
            .thenReturn(Optional.empty());
        
        // When & Then
        assertThrows(ProductNotFoundException.class, 
            () -> productService.getById(nonExistentId));
    }
}
```

## Best Practices

<CardGroup cols={2}>
  <Card title="Interface Segregation" icon="puzzle-piece">
    Separate interfaces for different operations (Register vs Get) allows clients to depend only on what they need.
  </Card>
  
  <Card title="Dependency Inversion" icon="arrows-rotate">
    Use cases depend on repository abstractions, not concrete implementations. This enables testing and flexibility.
  </Card>
  
  <Card title="Single Responsibility" icon="bullseye">
    Each use case has one clear business purpose. Complex operations can be composed from multiple use cases.
  </Card>
  
  <Card title="Domain Exceptions" icon="triangle-exclamation">
    Use custom domain exceptions (like ProductNotFoundException) rather than generic exceptions for better error handling.
  </Card>
</CardGroup>
